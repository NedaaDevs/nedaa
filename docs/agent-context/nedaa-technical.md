# Nedaa — Technical Briefing for AI Agents

> **Purpose of this file.** This is the canonical engineering context for Nedaa. Agents should read this when planning a new feature, scoping work, or making architectural suggestions, so they (a) don't re-discover what already exists and (b) reuse established patterns instead of inventing parallel ones. Pair with `nedaa-product.md` for product/marketing context.
>
> This file is **deliberately not a code reference**. It points at where things live; the code is the source of truth for _how_ things work.

---

## Stack snapshot

- **Runtime:** Expo SDK 57, React Native 0.86, React 19.2, **New Architecture mandatory**.
- **Language:** TypeScript strict. Path alias `@/*` → `./src/*` (always use the alias, not relative `./` paths).
- **UI:** Tamagui 2.1.0 (all packages pinned exactly, bumped as one family) + Moti animations. Custom config in `tamagui.config.ts`. Locale-aware font swap (Asap for Latin, IBM Plex Sans Arabic for Arabic).
- **State:** Zustand 5 with `persist` middleware, one store per domain in `src/stores/`.
- **Routing:** Expo Router 6, file-based, in `src/app/`. Tabs + nested settings sub-routes. Typed routes enabled.
- **Local persistence:** `expo-sqlite`. Per-domain DB service files in `src/services/` (one per feature: athkar, qada, umrah, hisn-muslim, quran).
- **Audio:** `react-native-nitro-player` — one global player for everything that makes sound (athkar sessions, Quran recitation, short sound previews). Access goes through `services/audio/nitroSession.ts`, a process-wide arbiter that hands ownership between callers so two features never drive the player at once.
- **Localization:** `i18next` + `react-i18next`. Locales: **en, ar, ms, ur**. Full RTL support. `expo-localization` for device locale detection.
- **Diagnostics:** on-device file logger (`src/utils/appLogger.ts`) — per-domain `.log` files, 30-day + 5 MB retention, JS crash capture, no remote telemetry. Logs stay on the device and are shared only when the user explicitly does so. (No Sentry — removed 2026-06.)
- **Validation:** `zod` for schema parsing.
- **Date:** `date-fns` + `date-fns-tz`. Hijri via `hijri-native`.
- **Package manager:** **bun** (not npm/yarn). `bun install`, `bun start`, etc.
- **React Compiler:** enabled (`experiments.reactCompiler: true`).

For exact pinned versions, read `package.json` — do not duplicate version numbers in this doc.

## Platforms & build variants

- **iOS** — bundle `dev.nedaa.app`. App group `group.dev.nedaa.app` for widget data sharing.
- **Android (GMS)** — package `dev.nedaa.android`. Default build.
- **Android (HMS source profile)** — same package and separate EAS profile `production-hms`. `BUILD_VARIANT=hms` conditionally selects in-tree HMS replacements and rejects Google Mobile Services runtime dependency groups. This is source-level architecture; release-artifact, physical no-GMS device, and AppGallery distribution validation are pending.
- **iOS Widget** — `ios/NedaaWidget/` (SwiftUI). Reads from a shared SQLite file in the app group.
- **No web** — `expo start --web` exists but is not a shipping target.

## Top-level directory map

```
src/
  app/            Expo Router screens. (tabs)/ is the tab nav; settings/, umrah/, custom-athkar/ are nested stacks.
  components/     UI components. Sub-folders by feature: athkar/, alarm/, quran/, umrah/, hijri/, onboarding/, feedback/, tools/, ui/.
  stores/         Zustand stores, one per domain (alarm, prayerTimes, athkar, notification, location, qada, umrahGuide, ...).
  services/       Business logic + DB access (db.ts, athkar-db.ts, athkar-player.ts, qada-db.ts, umrah-db.ts, ...).
  hooks/          Custom hooks (useCompass, useHaptic, useAlarmScreen, ...).
  utils/          Pure utilities (scheduling, notifications, date, compass).
  enums/          TypeScript enums. ALWAYS prefer enums over string literals in comparisons.
  types/          Shared type definitions.
  localization/   i18n config + locale JSON (en, ar, ms, ur).
  tasks/          Background task handlers (backgroundRefresh.ts).
  constants/      App constants (Alarm, Colors, Notification, AthkarAudio, ...).
  contexts/       React contexts.
  config/         App-level config.
  api/            Remote API clients.
  adapters/       Adapter-pattern integrations.
modules/          Custom local Expo native modules (see Capability Map).
ios/NedaaWidget/  SwiftUI widget extension.
android/          Android shell.
docs/             Documentation. agent-context/ holds this file; DEV-README.md covers developer setup.
plugins/          Custom Expo config plugins.
scripts/          Build/clean scripts (clean-all, nuke, build-local.ts, ...).
fastlane/         App Store / Play Store automation.
```

Ignore for code search: `node_modules/`, `dist/`, `builds/`, `ios/build/`, `android/build/`, `android/app/build/`.

## Cross-cutting conventions

- **Use enums from `src/enums/`** in comparisons — never compare against string literals (e.g., `ReaderViewMode.MADINAH`, not `"madinah"`).
- **Path alias `@/*`** is mandatory; do not use relative `./` imports.
- **No comments addressed to a person** ("// for Fahad", "// fixes the issue from yesterday"). General developer comments are fine where the _why_ is non-obvious.
- **One Zustand store per domain.** Persist middleware is the default for anything that should survive app restarts.
- **One DB service per domain.** Don't expand `db.ts` into a god-file.
- **Native modules auto-link via `expo-module.config.json`**, NOT via `app.json` plugins. (Plugins listed in `app.json` are the _Expo config plugins_, a different mechanism.)
- **Accessibility is a first-class requirement.** Every interactive element gets `accessibilityRole`, `accessibilityLabel` (via `t("a11y.*")` for user-facing screens), and `accessibilityState` where applicable. 44×44pt minimum touch targets. See `AGENTS.md` for the full checklist.
- **Tab UI pattern:** pill-shaped `Pressable` in an `HStack`. See `components/athkar/AthkarTabs.tsx` for the canonical example.

## External services & integrations

- **Crowdin** — community translations, project `nedaa-v2`.
- **CDN** — `cdn.nedaa.dev`, hosts athkar audio and **all Quran assets** (the shared content DB `quran.db`, per-edition mushaf image + meta/bounds bundles, ornaments, per-surah recitation bundles) served via the schema-2 `/quran/manifest`.
- **Backend** — separate repo `api/` (Elysia/Bun). Hosts the Quran manifest endpoint and (legacy) prayer-time endpoints.
- **Aladhan** — third-party prayer-times API; currently the only implemented provider, behind a provider abstraction.
- **EAS** — Expo Application Services for builds. Local builds use `scripts/build-local.ts`. HMS uses the `production-hms` profile.
- **Fastlane** — store submission automation under `fastlane/`.

## Build & release notes

- EAS Build packages files via git; `.gitignore` patterns can silently exclude native module files. Use `modules/*/android/build/` (with trailing slash), not `modules/*/android/build*` — the latter accidentally matches `build.gradle`.
- The `production-hms` build copies `agconnect-services.json` from a secret env var during `eas-build-pre-install`.
- `@react-navigation/{bottom-tabs,native,native-stack}` are **pinned exactly**. `expo install --fix` will loosen them — re-pin after running it. 7.2.x of `@react-navigation/native` breaks the custom `tabBar` render prop with a "Couldn't find a theme" error.
- Tamagui's 6 packages move as one atomic family; bump them together.

## What does NOT exist (so agents stop suggesting it)

- No user accounts, login, cloud sync, or server-side user state.
- No ads SDK. No analytics SDK. No crash/telemetry SDK (no Sentry) — diagnostics are on-device only.
- No paid tier, no in-app purchases, no entitlements.
- No web app target.
- No social/feed/community features.
- No donation flow.
- No native Android Wear or watchOS app (yet).
- No push notifications via a server — all notifications are scheduled locally.

## When planning a new feature, do this in order

1. Check the **Capability Map** above for an existing system to mirror.
2. Check `src/stores/` and `src/services/` for an existing domain that could absorb the work.
3. If new top-level domain: create one store, one service file (with DB if persistent), one components sub-folder, one route folder under `src/app/`, and entries in all four locale files.
4. If a new native capability is needed: scaffold a local module under `modules/` rather than reaching for a third-party package — the project favors maintained-in-tree native modules.
5. If a new screen has interactive elements: apply the accessibility checklist from `AGENTS.md`.
6. Use `enums/` from day one — don't ship string literals you'll have to refactor later.
